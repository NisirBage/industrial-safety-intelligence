# Judge FAQ

Written for the first Q&A round after a Presentation Mode or Judge
Mode walkthrough. Every claim below is grounded in something already
verifiable in this repository - a test, a source file, or a
documented limitation - not asserted from memory. Where a fuller
answer already exists elsewhere, this points to it rather than
duplicating it: `docs/demo/faq.md` (product-level FAQ), `docs/demo/
common_questions.md` (sharper technical cross-examination), and
`docs/architecture/CORE_FREEZE.md` (the authoritative engine spec).

---

## Architecture decisions

**Why deterministic instead of an LLM or a trained model?**
Industrial safety decisions need to be auditable and reproducible: a
regulator or incident investigator must be able to re-run the exact
inputs from six months ago and get the exact same tier back. An LLM or
a trained model can't give that guarantee cheaply, and "the model felt
like it was critical" is not an acceptable answer during an incident
review. This platform's whole bet is that a transparent, closed-form
formula gets real value before reaching for a black box. It's tested,
not just asserted: `tests/unit/test_no_wallclock_calls.py` statically
scans `src/domain` and `src/services` for wall-clock/`random.*` calls
and fails the build if it finds one; every agent, Fusion, Tiering, and
the full Risk Pipeline have dedicated tests that run identical inputs
twice and assert byte-identical output.

**Why PostgreSQL?**
Chosen at M1 for relational integrity across zones, sensors, permits,
equipment, and workers, with room to grow into time-series-specific
storage later without a migration. The Docker Compose stack pulls the
TimescaleDB image (`timescale/timescaledb:latest-pg16`) specifically so
sensor-reading hypertables are a configuration change, not a rewrite,
if reading volume ever demands it - not yet exercised in this sandbox,
and disclosed as such rather than claimed.

**Why replay/Time Machine instead of only a live dashboard?**
Because "what did the system know, and when" is exactly the question
an incident investigation or an auditor asks. `ReplayContext` is the
one shared state engine behind Time Machine, Digital Twin, Operations
Center, and Presentation Mode - scrub once, and every synchronized
view (map, decision graph, action queue, executive summary) jumps to
that same tick. It's the same mechanism a live sensor feed would use
once one exists; nothing about replay is a separate code path from
production.

**Why a Digital Twin instead of a table of numbers?**
A shift supervisor scanning for what needs attention right now reads a
spatial layout faster than a sortable table. Every icon on it is real
data (worker counts, active permit types, sensor gas types, tier
color) - the site-plan layout itself is a fixed table of five named
zones, a disclosed limitation, not a claim of infinite scale.

**Why an Operations Center distinct from the Executive Dashboard?**
Different audiences, same underlying data. Operations Center answers
"what do I do right now, in what order" for a shift supervisor
(prioritized action queue, dependencies, SOP references, ETA). The
Executive Dashboard answers "is the plant safe to operate" for a plant
manager in one glance (Safety Status, Operational Readiness, Current
Incident). Both are pure derivations over the same `RiskAssessment`
and `RiskJustification` data - neither computes a new risk value.

**Why polling instead of WebSockets for live updates?**
Simplicity and testability over premature optimization: `PollingContext`
re-fetches on a fixed interval, which is trivial to test, trivial to
reason about, and fast enough that judges never notice the difference
in a demo. A push-based channel is listed as a legitimate future
improvement (see Roadmap below), not a hidden gap.

## Likely technical questions

**Is any of this actually live, or is it a fixed demo?**
Live. Real FastAPI backend, real PostgreSQL, real deterministic
pipeline. The only synthetic input is the sensor stream itself,
generated by an authored scenario YAML and run through
`risk_pipeline.py` - the exact same code path a real sensor feed would
use. Nothing in the UI is a hardcoded number; the Demo Readiness panel
on the Presentation Mode launcher proves this live, per run, by
actually calling `GET /api/v1/health`, `GET /api/v1/scenarios`, and
`GET /api/v1/risk/current` and reporting exactly what's missing if
anything is down.

**How do I know the frozen engine hasn't quietly changed?**
Verified via git history, not a comment: every frozen module's current
content was diffed against its original milestone commit and confirmed
byte-identical as part of an independent audit before UI-focused
milestones began. `docs/architecture/CORE_FREEZE.md` §3 lists every
frozen file explicitly; §10/§11 define what would and wouldn't require
an ADR to change.

**What stops the "Why" explanation from being generated after the
fact to match the score?**
It isn't generated at all - every justification sentence is templated
from fields the deterministic engine already computed and persisted
(a fired rule identifier, an interaction-bonus value, a sensor-to-
threshold ratio). No LLM call, no free text generation, anywhere in
that path.

**What stops the counterfactual comparison from secretly sharing logic
with the real engine, making the comparison meaningless?**
A permanent structural test enforces it: the Counterfactual Comparator
is asserted to never import, call, or otherwise share code with the
Scheduler, Fusion, Tiering, Justification Builder, or any agent module.
Independent by construction, not by convention.

**Are the "impact" / "business impact" numbers on Operations Center
and Presentation Mode real projections?**
No, and this is deliberate: they are categorical labels (CRITICAL /
VERY HIGH / HIGH / MODERATE / LOW / INFORMATIONAL, or a plain
tier-to-impact mapping), never a fabricated number like "Expected Risk:
94 -> 37." No API exposes Fusion's real per-agent weights or a "what if
this agent were removed" branch, so any such number would be invented.
Where the UI shows a percentage (e.g. the Decision Graph's per-agent
share), it's explicitly labeled as a normalization of real numbers
(`risk_i / sum(risk_j) x 100`), never claimed to be the literal Fusion
weight.

**What's the actual test coverage?**
467 automated tests: 294 backend (`pytest`, against a live PostgreSQL
instance) and 173 frontend (`vitest`). Backend `ruff`, `black --check`,
and `mypy --strict` are clean; frontend `oxlint` and `tsc -b` are
clean. 16 of the frontend tests currently fail for one disclosed,
non-architectural reason (see Limitations below) - not silently hidden.

## Scalability

**Does this scale past a handful of zones?**
The underlying data model has no zone-count limit - zones, sensors,
permits, and workers are ordinary relational rows. The one place a
fixed count exists is the Digital Twin's visual site-plan layout
(currently five named zones, laid out by hand) - a labeled, disclosed
limitation, not a hidden one. Generalizing the layout to an arbitrary
zone count is a frontend visualization task, not an engine change.

**Does the risk pipeline itself scale?**
Each zone-tick computation is O(1) in the number of zones (four
independent agents + one fusion step per zone, no cross-zone
coupling), so horizontal scaling is adding zones, not slowing down
existing ones. `scripts/benchmark_deterministic_engine.py` measures
this directly against the frozen engine.

**Does the database scale?**
PostgreSQL via TimescaleDB is chosen specifically so sensor-reading
tables can become hypertables (time-partitioned) without an
application-level rewrite - a configuration change already anticipated
in the Compose stack, not yet exercised against production-scale data
volume in this sandbox.

## Roadmap

Honest "what's next," not a wishlist:

1. **Push-based live updates** (WebSocket/SSE) to replace polling for
   sub-second latency at the cost of more moving parts - deferred
   because polling is simpler, testable, and fast enough for this
   platform's actual update cadence.
2. **A generalized plant-layout renderer** for the Digital Twin, so an
   arbitrary zone count/shape doesn't require hand-authoring SVG
   coordinates.
3. **Multi-gas-per-zone support** in the Gas Risk agent and
   Counterfactual Comparator (currently assume at most one monitored
   gas type per zone - see Limitations).
4. **A real persistence store for `AgentCache`/`TierState`** across
   process restarts (currently in-memory).
5. Fix the 16 known frontend test failures by aligning the mock
   server's port with `VITE_API_BASE_URL` (see Limitations) - a test
   fixture fix, not a product change.

None of these touch the frozen engine's algorithms; all of them would
*consume* it through the same contracts documented in
`docs/architecture/CORE_FREEZE.md` §6.

## Limitations (disclosed, not discovered by judges)

- **Gas Risk and the Counterfactual Comparator assume at most one
  monitored gas type per zone.** Matches today's seed data exactly but
  is unenforced by the schema - documented in `CORE_FREEZE.md` §12.
- **The Digital Twin's site-plan layout is a fixed table of five named
  zones.** The data model itself has no such limit.
- **`AgentCache`/`TierState` have no persistence store across process
  restarts.** In-memory only today.
- **`uuid.uuid4()` remains the SQLAlchemy column-level default on every
  ORM model's primary key.** Inert in practice (every real call site
  supplies a deterministic id explicitly) but latent: a future caller
  that omits an explicit id would silently get a random one.
- **16 of 173 frontend `vitest` tests currently fail** because the
  mock server (MSW) hardcodes `localhost:8000` while
  `VITE_API_BASE_URL` resolves to the actual dev port (`:8010`) in this
  environment - a test-fixture/environment mismatch, not a defect in
  the code under test. All 294 backend `pytest` tests pass.
- **This is a decision-support and explainability layer, not a
  certified Safety Instrumented System (SIS).** It is meant to sit
  alongside existing hard-wired safety interlocks, not replace them.
