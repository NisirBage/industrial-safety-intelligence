# Presentation Mode Demo Script

This covers the **auto-advancing, ~90-second guided tour** at `/story`
(`PresentationModePage.tsx`) - a distinct artifact from the manual
7-minute/3-minute walkthroughs in `docs/demo/demo_script.md`, which
remain the right choice when there's time for questions mid-tour.
Presentation Mode is for the first 90 seconds of a judging slot, or as
a hands-free loop at a booth.

Click **Start Demo**. Everything below happens automatically - no
clicking required unless you want to pause, skip, or go back.

Every value narrated below is a live number the platform actually
computed for whichever real scenario `lib/presentationScript.ts::
selectPresentationScenario` picks from the catalog (it prefers one
whose name mentions "SIMOPS" or "critical" - today that's the Tank
Farm SIMOPS Conflict scenario, but the script below is written so it
stays true even if the catalog changes and a different scenario gets
picked).

---

## Scene 1 — Industrial Safety Intelligence (3s)

**Expected screen:** Full-screen title, fading in. Plant status badge,
zone count, active permit count.

**Narration:** "This is Industrial Safety Intelligence — a
deterministic, explainable early-warning platform. Right now it's
watching [N] zones with [N] active permits."

**Judge talking point:** Every number on this slide is live -
`GET /zones`, `GET /risk/current`, `GET /permits`, nothing scripted.

**Likely question:** "Is this a real deployment or a demo shell?"
**Suggested answer:** "Real backend, real Postgres, real deterministic
engine — the only thing simulated is the sensor input, generated by
an authored scenario file and run through the exact same pipeline a
real sensor feed would use."

---

## Scene 2 — Live Digital Twin (8s)

**Expected screen:** The interactive plant map — zone shapes, worker
badges, gas-sensor glyphs, permit-type icons, tier coloring.

**Narration:** "Here's the plant as a digital twin — not a static
diagram. Every icon is real: who's in a zone, what permits are active,
which sensors are reporting."

**Judge talking point:** Permit icons are shape-coded by real permit
type (flame for Hot Work, hatch for Confined Space) — point at one if
active.

**Likely question:** "Does this scale past a handful of zones?"
**Suggested answer:** "The site-plan layout is currently a fixed table
of five named zones — disclosed as a known limitation, not hidden.
The underlying data model has no such limit; the visual layout is the
part that would need generalizing."

---

## Scene 3 — Incident Detected (10s)

**Expected screen:** A single zone/tier/score card at the exact tick
the replayed scenario first leaves NORMAL.

**Narration:** "Watch what happens when a real incident starts. This
is the first real tick where risk leaves normal — not a mocked
alert."

**Judge talking point:** This tick comes from `lib/presentationScript.ts::
findFirstEscalationIndex` — the first index in the real replay
history whose tier isn't `normal`, never a hand-picked moment.

---

## Scene 4 — AI Decision Engine (12s)

**Expected screen:** The Decision Graph pipeline diagram — Sensors →
Context Builders → four agents → Fusion → Tiering → Recommendations —
at the scenario's single highest-risk tick.

**Narration:** "Four independent deterministic agents — gas risk,
worker exposure, equipment status, permit intelligence — each produce
a 0-100 risk score. Fusion combines them. This is not a black box —
click any node and you see the exact rule that fired."

**Judge talking point:** "Deterministic" is a hard claim — invite them
to ask "what if I ran this scenario twice?" (same inputs, byte-
identical output, proven by the project's own determinism test suite).

**Likely question:** "Where's the AI/ML?"
**Suggested answer:** "There isn't one in the decision path, by design.
Every score here is a documented closed-form formula — a saturating
curve, a weighted sum, a hysteresis gate. AI/LLM use in this platform
is scoped strictly to narrative explanation, never risk computation."

---

## Scene 5 — Decision Graph (10s)

**Expected screen:** Ranked agent contributions with each agent's
share of this tick's total raw risk, the top contributor called out,
and whether an interaction bonus applied.

**Narration:** "Here's why the score is what it is. [Agent] is the
largest contributor at [X]% of this tick's raw risk. [If applicable:]
An interaction bonus is applying right now — multiple independent
factors are compounding at once, something a single-sensor alarm
would never catch."

**Judge talking point:** The percentage shown is a relative share of
real numbers, explicitly labeled as such — not the literal internal
Fusion weight (which isn't exposed by any endpoint and is never
guessed at).

---

## Scene 6 — Operations Center (12s)

**Expected screen:** Action Queue (top action auto-expanded),
Operational Dependency Graph, Impact Explorer, Operator Timeline.

**Narration:** "The platform doesn't just detect — it recommends.
Here's the prioritized action queue: what to do, in what order, with
which dependencies, mapped to a real SOP reference."

**Judge talking point:** Point at the Impact Explorer's qualitative
labels (CRITICAL/HIGH/etc.) — deliberately never a fabricated "risk
drops from X to Y" number, since no persisted computation produces
one; call this out as an intentional honesty choice, not a gap.

---

## Scene 7 — Executive Dashboard (8s)

**Expected screen:** Animated counters for current risk, business
impact (qualitative), and operational readiness, plus the same
deterministic one-paragraph executive summary from Scene 5's data.

**Narration:** "For a plant manager, one screen: is the plant ready to
operate? Right now: [readiness label]."

---

## Scene 8 — Counterfactual (10s)

**Expected screen:** Three cards — Current vs naive baseline (real,
computed comparison), Without Permit (qualitative), Equipment Restored
(qualitative).

**Narration:** "The naive baseline — a single-sensor threshold alarm —
[agrees/misses] this escalation. [If a miss:] This is the platform's
core value proposition: catching what a legacy system doesn't."

**Judge talking point:** "Without Permit"/"Equipment Restored" are
explicitly qualitative — direction of change, never a computed score,
because the deterministic engine has no per-agent "what if" branch.
This is disclosed, not hidden.

---

## Scene 9 — Time Machine Replay (10s)

**Expected screen:** The real Time Machine replay controller, playing.

**Narration:** "Every tick you just saw is scrubbable — an operator or
auditor can replay any historical incident, tick by tick, from
persisted data alone."

---

## Scene 10 — Closing (8s)

**Expected screen:** Animated platform-statistic counters, then the
closing tagline.

**Narration:** "4 deterministic agents. 21 distinct rule identifiers.
19 REST endpoints. 21 frontend pages. 467 automated tests. Dockerized,
monitored with Prometheus, documented with OpenAPI, backed by
PostgreSQL, structured logging throughout. Deterministic. Explainable.
Production ready."

**Likely question:** "How do I know those numbers are real?"
**Suggested answer:** "Every one of them is grep-verified against this
repository at the time this feature was built — see
`lib/presentationScript.ts::PLATFORM_STATS`'s own docstring for the
exact command used to count each one."

---

## Controls reference

- **▶ Start Demo** — begins at Scene 1, selects and starts the replay scenario.
- **⏸ Pause / ▶ Auto Play** — freezes/resumes the scene timer (resumes from wherever it paused, not from the top of the scene).
- **⏭ Next / ⏮ Previous** — manual scene navigation, for skipping ahead if a judge asks a question mid-tour.
- **Restart** — returns to Scene 1 and resets the replay cursor.
- **`D`** — toggles the debug HUD (scene, elapsed/remaining time, FPS, replay tick, current zone) - useful for the presenter, not intended for the judge-facing view.
