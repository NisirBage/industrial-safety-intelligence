# FAQ

**What is this?**
An industrial safety decision-support platform. It ingests sensor
readings, permit activity, worker locations, and equipment status for
a plant, computes a compound risk score per zone using four
independent reasoning agents, and explains every decision in plain
language - with a side-by-side comparison against what a traditional
single-sensor alarm system would have concluded at the same moment.

**Who is it for?**
Plant safety officers and shift supervisors who need a real-time,
plant-wide view of risk; safety engineers who need to audit *why* a
score was what it was; and, on the demonstration side, anyone
evaluating whether "explainable, deterministic decision intelligence"
is a credible alternative to either a dumb threshold alarm or an
opaque black-box model.

**Is the risk score real, or a mock?**
Real. Every number on every page traces back to a deterministic
computation over persisted sensor/permit/equipment/worker data,
served by a live FastAPI backend backed by a live PostgreSQL database.
There is no hardcoded demo number anywhere in the UI.

**What does "deterministic" mean here, concretely?**
Identical inputs (including identical database state) always produce
identical outputs, at every stage of the pipeline. No `random.*` call
exists anywhere in the backend's source; every entity/scenario id is
derived from a deterministic hash, never generated randomly; there is
no wall-clock read inside the risk computation itself.

**What's a "zone"?**
A physical area of the plant with its own sensors, e.g. "Tank Farm" or
"Compressor House." Risk is computed and tracked per zone.

**What's the difference between the "compound engine" and the "naive
baseline"?**
The compound engine is this platform's own reasoning: four independent
agents, fused with cross-signal interaction awareness, mapped to a
tier through hysteresis. The naive baseline is a traditional
single-sensor threshold alarm - a hard trip point per sensor. Both are
shown side by side on purpose, because the entire value proposition is
in the gap between them.

**Can I trust the "Why" explanation, or is it generated by an LLM?**
No LLM anywhere. Every "why" sentence is templated from fields the
deterministic engine already computed and persisted (an interaction
bonus value, a sensor-to-threshold ratio, a fired rule identifier) -
never freely generated text.

**What happens if a sensor goes offline?**
The relevant agent falls back to a documented fail-safe behavior
(visible in its own rules_fired, e.g. `missing_data_fail_safe` or
`stale_data_fail_safe`) rather than silently defaulting to "safe."

**Does this replace a certified safety instrumented system (SIS)?**
No - this is a decision-support and explainability layer, not a
safety-rated interlock system. It is meant to sit alongside existing
hard-wired safety systems, not replace them.
