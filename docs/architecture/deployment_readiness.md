# Deployment Readiness Report

Production-deployment audit. Documentation only where noted; a small,
disclosed set of code changes closes the genuine gaps this audit found
(listed in full under "Fixes applied" below). No file under
`src/domain/**` was modified — the deterministic engine is unaffected.

## Architecture (as actually implemented, not as originally planned)

| Layer | Technology | Evidence |
|---|---|---|
| Frontend | Vite + React 19 + TypeScript + TanStack Query | `frontend/package.json`, `frontend/vite.config.ts` |
| Backend | FastAPI (Python 3.12), Uvicorn | `src/api/main.py`, `deploy/Dockerfile.backend` |
| Database | PostgreSQL (TimescaleDB optional, auto-detected) | `src/infra/db/migrations/versions/0001_initial_schema.py` |

| Graph database | **None** | The "Knowledge Graph" feature (`src/knowledge_graph/service.py`) is request-time, in-process Python traversal (bounded BFS, `_MAX_SUBGRAPH_NODES = 60`) over ordinary PostgreSQL query results via existing repositories — not a stored graph, not Neo4j, not a graph query language. |
| Authentication | **None** | No `jwt`/`oauth`/`passlib`/`fastapi.security` anywhere; every endpoint (including the two write endpoints, `/scenario-builder/execute` and `/ingest/reading`) is unauthenticated today. See "Security" below. |

## Discrepancy between documentation and implementation (found, then resolved)

`docs/architecture/integration_readiness.md` and `deploy/docker-compose.yml`'s own (now-stale) header comment described Redis and Chroma as forthcoming M7/M11 dependencies. Neither milestone was ever built. Implementation (zero imports, zero client libraries in `requirements.txt`) is the source of truth, per this audit's own instruction. `deploy/docker-compose.prod.yml` had already independently reached the same conclusion and excluded both services with an explicit comment explaining why — `deploy/docker-compose.yml` (dev) had not yet been brought in line with that same decision. **Fixed**: removed both services (and the backend's `depends_on` on them) from the dev compose file, matching the production compose file's already-correct, already-documented decision. This is a pure simplification — no functionality was removed, since none existed.

## Backend audit

- **Runtime**: Python 3.12, Uvicorn, single process, no multi-worker config (`CMD ["uvicorn", "src.api.main:app", "--host", "0.0.0.0", "--port", "8000"]` in `deploy/Dockerfile.backend`).
- **Dependencies**: `requirements.txt` — FastAPI, SQLAlchemy, psycopg (v3), Alembic, pydantic-settings, prometheus-client. No LLM, vector-DB, graph-DB, or object-storage client. `requirements-dev.txt` adds ruff/black/mypy/pytest.
- **Startup process**: `alembic upgrade head` → (optional) `python -m src.infra.db.seed` → (optional) scenario replay → `uvicorn src.api.main:app`. No startup script wraps these into one command today (a genuine gap for a one-command deploy — see DEPLOYMENT.md's own entrypoint recommendation).
- **Health endpoints**: `/api/v1/health` (existing, DB+migration check), `/api/v1/health/platform` (existing, per-subsystem dashboard). **Added this audit**: `/api/v1/live` (process-only, never checks the database — a true liveness probe) and `/api/v1/ready` (same check as `/health`, added as its own documented route so the conventional `/health` + `/ready` + `/live` trio is complete).
- **Migrations**: Alembic, hand-authored (not autogenerated). `0001` now detects TimescaleDB availability via `pg_available_extensions` and only converts tables to hypertables when the extension is genuinely installed — verified against a real PostgreSQL 17 instance with no TimescaleDB, `alembic upgrade head` succeeds (see the TimescaleDB portability work earlier in this project's history for the full validation).
- **Seed process**: `src/infra/db/seed.py::seed()` — idempotent (deterministic UUIDs + `merge()`), verified by direct re-run.
- **Logging**: structured JSON via `src/config/logging.py::JSONFormatter`, one line per log record, stdout only (container-friendly, no file handler). **Fixed this audit**: added an always-present `timestamp` field (previously absent) and a `request_id` correlation field (previously no request-ID mechanism existed at all).
- **Configuration**: single `Settings` object (`src/config/settings.py`), `pydantic-settings`, reads `.env`. **Fixed this audit**: `database_url` now has a `field_validator` that fails at startup with a clear message on a malformed DSN or non-PostgreSQL driver, rather than deferring to an opaque failure at the first query.
- **Secrets**: `.env` is gitignored; `.env.example` documents every variable with no real secret committed.
- **CORS**: explicit origin allow-list (never `"*"`), `allow_credentials=True` made safe by that explicit list — already correct, unchanged.
- **Middleware**: CORS, then a combined request-logging + Prometheus-metrics + (as of this audit) request-ID middleware; three global exception handlers (`APIError`, `RequestValidationError`, catch-all `Exception`) producing one error envelope shape, never leaking a stack trace to the client — already correct, unchanged.

## Frontend audit

- **Build process**: Vite. `npm run build` runs `tsc -b && vite build` (type-check, then bundle).
- **Production build**: static assets to `frontend/dist/`, served by nginx in `deploy/docker-compose.prod.yml` (`deploy/Dockerfile.frontend`, multi-stage: `node:20-alpine` build stage → `nginx:1.27-alpine` runtime stage).
- **Environment variables**: exactly one, `VITE_API_BASE_URL` (`frontend/src/api/client.ts`), defaulting to `http://localhost:8000` for local dev and built as an empty string (`""`) in the production image so the browser calls same-origin, relative `/api/v1/...` URLs.
- **API routing**: nginx reverse-proxies `/api/` to the `backend` service by its Compose DNS name (`deploy/nginx/nginx.conf`) — no CORS needed in the production topology since frontend and API share one origin.
- **Static assets**: served directly by nginx; client-side routing handled by `try_files $uri $uri/ /index.html` so deep links survive a hard refresh.

## Infrastructure dependencies (what a fresh deploy genuinely needs)

**Required**: PostgreSQL 13+ (any standard installation — TimescaleDB is detected automatically if present, never required; see the project's own TimescaleDB portability work).

**Not required, and not shipped in the production compose file**: Redis, Chroma/any vector DB, Neo4j/any graph DB, object storage (S3/MinIO), any LLM API key. None of these are consumed by any code path that actually runs.

## Security review (genuine findings only)

1. **No authentication or authorization on any endpoint** (High — the single largest gap for anything beyond a hackathon demo). Every read endpoint and both write endpoints (`POST /scenario-builder/execute`, `POST /ingest/reading`) are open to any caller who can reach the API. **Not fixed in this pass** — adding real authentication is a genuine new feature (user model, session/token issuance, a login UI) that changes the public API contract, which this audit's own engineering principles explicitly say to preserve "unless a deployment blocker requires a change." Lack of auth is not itself a deployment blocker (many hackathon demos run this way deliberately) — it is a documented, accepted risk for the demo, and the top item under "Remaining Technical Debt" for any future production use beyond a demo.
2. **CORS** — already a proper explicit allow-list; no finding.
3. **No rate limiting** (Medium). No `slowapi`/token-bucket middleware exists. Combined with #1, a public deployment's write endpoints have no abuse protection. Documented as technical debt, not fixed here (adding a new dependency + middleware for a demo-scale deployment was judged out of proportion to the actual risk at hackathon scale).
4. **File uploads / path traversal**: `csv_watcher.py` accepts an arbitrary filesystem path, but is deliberately never exposed over HTTP (`src/api/routers/ingest.py`'s own docstring states this explicitly) — no finding, already correctly scoped.
5. **SQL injection**: all queries go through SQLAlchemy's ORM/Core with bound parameters; the only raw SQL (`src/infra/db/migrations/versions/0001_initial_schema.py`'s TimescaleDB helper functions) interpolates only hardcoded, developer-authored table/column name literals, never request input — no finding.
6. **XSS/CSRF**: React escapes all rendered content by default; no `dangerouslySetInnerHTML` found in a targeted grep. CSRF is not applicable to a stateless, cookie-free, credential-less REST API with no session cookie — no finding.
7. **HTTPS**: not terminated by this application (nginx serves plain HTTP on port 80 in `docker-compose.prod.yml`); expected to be terminated by whatever sits in front in a real deployment (a managed platform's own edge, or an added Let's Encrypt/Caddy layer) — documented in DEPLOYMENT.md, not a code-level finding.
8. **Environment variables / secrets**: `.env` correctly gitignored; no secret found hardcoded in `src/` or `frontend/src/`.

## Performance review

- **Frontend bundle**: code-split by route (`React.lazy` + `Suspense`, confirmed across `App.tsx`'s route definitions from earlier milestones); no single-bundle regression found.
- **Backend startup**: sub-second for the FastAPI app itself; the dominant real-world startup cost is the database connection + migration check, not application code.
- **Database connections**: SQLAlchemy's default connection pool (no explicit pool-size tuning) — reasonable for a single-instance hackathon deployment; a real production deployment under concurrent load should tune `pool_size`/`max_overflow` explicitly, noted as technical debt rather than fixed here (no measured bottleneck exists to tune against).
- **No performance regression was measured, because none was found** — this audit did not identify a measured, real bottleneck to optimize, per the standing instruction to optimize only measured problems, not hypothetical ones.

## Fixes applied (full list)

1. `.dockerignore` created (did not exist — the entire repo, including `.venv`, `.git`, `node_modules`, and `.env` itself, was previously sent as Docker build context).
2. `src/config/settings.py` — `database_url` now validated at startup (fails fast on a malformed DSN or non-PostgreSQL driver).
3. `src/config/logging.py` — every log line now carries a `timestamp`; `request_id` added to the existing correlation-field mechanism.
4. `src/api/main.py` — request-ID middleware (reuses an incoming `X-Request-ID` header if present, generates one otherwise; echoed back as a response header; attached to the request's log line).
5. `src/api/routers/health.py` — added `/api/v1/live` and `/api/v1/ready`.
6. `deploy/docker-compose.yml` — removed the unused `redis`/`chroma` services and the backend's dependency on them.
7. `.github/workflows/ci.yml` — added a `docker` job that builds both production images (build-only, no registry push — this repo has no configured deployment credentials).
8. `DEPLOYMENT.md`, `ENVIRONMENT.md`, `TROUBLESHOOTING.md` created; `README.md` updated to reference them.

## What was not executed, and why

- **No live deployment to Railway/Render/Vercel/AWS was performed.** This environment has no cloud account credentials configured for this project; DEPLOYMENT.md documents the exact commands for a user who does.
- **`docker compose build`/`docker compose up` were not executed in this session** — Docker itself is not installed in this sandbox (confirmed: `docker` is not on `PATH`), consistent with this project's own long-standing, previously-documented limitation (`docs/architecture/integration_readiness.md`'s "Docker was never available in the sandbox that built this platform"). The Dockerfiles and compose files were reviewed statically instead; the CI `docker` job added in this pass will be the first real execution of `docker build` against these files, the next time CI runs on GitHub's own runners (which do have Docker).
- **No live authentication system was added** — see Security finding #1's own rationale.
